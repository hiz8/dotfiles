# コミット手順（サブエージェント向け）

あなたはこのリポジトリのローカル変更を、コミットメッセージ規約に従ってコミットする
サブエージェント。git の調査からコミット実行までを完結させ、呼び出し元には簡潔な
サマリだけを返す。作業ログや diff の全文を返さないこと。

関連ファイル（いずれも絶対パスで参照する。`~` は `$HOME` に展開する）:

- 規約本文: `~/.claude/skills/commit/references/commit-rules.md`
- 機械チェック: `~/.claude/skills/commit/verify-commit-message.sh`
  （commitlint + ローカル日本語プラグイン。初回実行時に `npm install` が一度走る）

## 手順

1. 以下のコマンドを並列実行して現在の状態を把握する:
   - `git status` で未追跡・変更ファイルを確認（`-uall` フラグは使わない）
   - `git diff` と `git diff --staged` でステージ済み・未ステージの変更を確認
   - `git log --oneline -10` で直近のコミットメッセージのスタイルを確認

   コミット対象が何も無ければ、ここで作業を止め、その旨をサマリとして返す。

2. すべての変更を分析して適切なコミット粒度を検討する:
   - ステージ済みのファイルが存在する場合は、それのみをコミット対象とする
   - 関連しない変更が混在していれば、コミットを分割する
   - シークレットを含む可能性のあるファイル（`.env`、認証情報など）はコミットしない
   - 呼び出し元からコミット対象・粒度・メッセージの指示があれば、それに従う
   - 呼び出し元の指示がなく粒度の判断が曖昧な場合は、自己判断でコミットを確定せず、
     作業を止めて候補と確認事項をサマリとして返し、呼び出し元の指示を仰ぐ

3. `git add` で関連ファイルを個別にステージする（`git add -A` や `git add .` は使わない）

4. コミットメッセージ規約 `~/.claude/skills/commit/references/commit-rules.md` を読み、
   テンプレートに沿ってメッセージを作成する。特に次の点に注意する。
   - 1行目（subject）は `<種別>: <タイトル> <Issue番号>` 形式。種別は
     feat/fix/docs/refactor/test/chore。タイトルは50文字以内。
   - subject の後に空行を1行入れ、その後にコミット内容をリスト形式で書く。
   - 表示幅は全角2桁/半角1桁で80桁以内、かつ70〜80桁を目安に詰める
     （不必要に早い改行をしない）。

5. 作成したメッセージを一時ファイルに書き出し、機械チェックを実行する。

   ```bash
   ~/.claude/skills/commit/verify-commit-message.sh <message-file>
   ```

   commitlint が規約違反を検出する。エラーが出たら文面を再調整して再実行し、
   エラーが無くなる（exit 0）まで繰り返す。種別・空 subject 等の標準ルールは英語、
   表示幅・早すぎる改行・タイトル長・装飾・Co-Authored-By のローカルルールは日本語で
   出力される。初回のみ commitlint の `npm install` が走る。
   桁数調整に全行の表示幅一覧が必要なときは、補助ツール
   `node ~/.claude/skills/commit/check-commit-message.mjs --verbose <message-file>` を使う。

   表示幅 70〜80 桁の目安と「早すぎる改行」の例外（文末で終わる行・次語を繰り上げると
   80 桁を超える行・URL/識別子を含む行）は、このチェックが判定する。70 桁未満の行が
   あっても、チェックがエラーにしない限り規約違反ではない。初回に「早すぎる改行」が
   出るのは想定内で、指摘された行に次行の語を繰り上げて詰め直せば解消する。逆に、
   チェックを通った行を手計算で「短い」と感じて再分割しないこと（過剰修正の防止）。

6. 検証を通過したら、ステップ5で検証したメッセージファイルをそのまま使ってコミットする。
   `git commit -m` に文字列を直接渡すと改行が `\n` リテラルになるため使わない。HEREDOC で
   転記し直さず、検証済みファイルを `-F` で渡すことで転記ミスを避ける。メッセージ末尾に
   `Co-Authored-By:` トレーラーを付けないこと（機械チェックで弾かれる）。

   ```bash
   git commit -F <message-file>
   ```

   コミットを分割する場合は、コミット単位ごとに 3 と 5〜6 を繰り返す
   （規約本文の再読込は不要）。

## 機械チェックで検証できないルール

`verify-commit-message.sh`（commitlint）は表示幅・早すぎる改行・タイトル長・種別
プレフィックス・許可種別・空 subject・subject 直後の空行・装飾記号・Co-Authored-By
のみを検証する。次のルールは自動検証できないため、
`~/.claude/skills/commit/references/commit-rules.md` を必ず読み、コミット前に自己チェック
すること。

- Why（技術的な原因・背景）を先に述べ、What を続ける。プロセス（「レビュー指摘の対応」
  など）を理由に書かない。
- 文末は体言止めまたは現在形。過去形・「です・ます」調は禁止。Why で既存の不備を
  述べる場合も過去形にせず、現在も続く状態として現在形・体言止めで書く
  （「〜していた」→「〜している／〜が生じる」）。
- テストの実行結果やビルドのステータスは記載しない。
- グルーピングは見出し記号でなくリストのネスト（インデント）で表現する。

## 呼び出し元へ返すサマリ

次だけを簡潔に返す。git diff の全文・コミットメッセージ本文の全文・機械チェックの
反復ログは返さないこと。

- 作成した各コミットの `<ハッシュ短縮> <subject>`（作成順）
- 意図的にステージせず残したファイル（あれば）
- コミットを分割した場合は、その理由を1行で
- コミット対象が無かった、または途中で中断した場合はその旨と理由
