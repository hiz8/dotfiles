# コミットメッセージ規約

SKILL.md とは分離し、ルールだけを参照できるようにしている。各ルールには末尾に検証区分を付す。

- [機械] … `check-commit-message.mjs` が自動検証する。
- [自己] … スクリプトでは検証できないため、人/エージェントが自己チェックする。

## テンプレート

```
<コミット種別>: <コミットタイトル> <Issue番号>

<コミット内容>
```

## ルール

### コミット種別

- 先頭にコミットの種別を示す文字列をコロンと半角スペース区切りで記載する。[機械]
  - feat
  - fix
  - docs
  - refactor
  - test
  - chore
- 種別の選び方（迷ったとき）。[自己]
  - feat … 利用者から見た新しい挙動・機能の追加
  - fix … 既存の不正・不足な挙動の是正（バグ修正）
  - refactor … 外部から見た挙動を変えない内部整理
  - docs … ドキュメントのみの変更
  - test … テストのみの追加・修正
  - chore … ビルド・依存・設定など上記以外
  - 性能改善は、挙動が変わるなら feat、変わらない整理なら refactor に寄せる。
    `perf` など上記リスト外の種別は使わない。

### コミットタイトル

- 50文字以内で簡潔に記述する。[機械]
- 文末は「体言止め（例：〜の修正）」または「現在形（例：〜を変更）」とし、過去形や
  「です・ます」調は禁止する。[自己]

### Issue番号

- コミットに直接関連する Issue の番号があれば、それを記載する。[自己]
  - 記法は subject 末尾の ` #42` を既定とする（body 先頭の `#42 の対応` も可）。
  - 関連 Issue が無ければ省略する（プレースホルダを残さない）。

### コミット内容

- リスト形式でコミットの内容を説明する。[自己]
- 1行の表示幅は半角80桁以内にすること。全角文字は2桁、半角文字は1桁として数える。[機械]
- 改行は「80桁を超える直前」で行い、80桁ぎりぎりまで詰めること。[機械]
- 80桁以内でも不必要に早い改行を禁止する。原則として、各行は表示幅70〜80桁を目安に
  詰める。[機械]
- ただし、以下は70桁未満でも許容する。[機械]
  - 文の終端で自然に終了する行
  - 次行に続けると80桁を超える行
  - URL・コマンド・識別子など分割不適切な文字列を含む行
  - 次行が新しいリスト項目（行頭が `-` `*` `+` または `数字.`）として始まる行。
    その行は折り返しの継続行ではないため、短くても詰め直し不要
- 改行位置は単語・助詞・読点の区切りを優先し、意味の切れ目を保つこと。[自己]
- コミット前に、全行について全角2桁/半角1桁の表示幅で自己チェックすること。[機械]
- 原則、Why（技術的な原因・背景）を先に述べ、What（具体的な変更点）を続ける。ただし、
  コードを見れば What が理解できる場合は What の記載を省略する。[自己]
  - Why には「技術的な原因」や「仕様上の不備」を記載すること。「レビューで指摘された
    ため」や「修正依頼の対応」といった、修正のきっかけ（プロセス）は記載しないこと。
- 特定の Issue に関するコミットであれば、先頭に `<Issue番号> の対応` と記載する。[自己]
- 文末は「体言止め（例：〜の修正）」または「現在形（例：〜を変更）」とし、過去形や
  「です・ます」調は禁止する。[自己]
  - Why で既存の不備を説明する場合も過去形にしない。問題は過去の出来事ではなく現在も
    続く状態として現在形・体言止めで書く（「〜していた」→「〜している」「〜が生じる」、
    「〜になっていた」→「〜になっている」）。
- Markdown記法による装飾（`**` による太字など）は禁止。すべてプレーンテキストで記述
  すること。[機械]
- Markdownの装飾（太字等）や、強調記号（`【】`, `[]`等）の使用は禁止する。[機械]
  （`**` `【` `】` は機械検証する。`[]` は `[CI]` 等の正当用例があるため自己チェック）
- 項目のグルーピングは、見出し記号を使わず「リストのネスト（インデント）」で表現
  すること。[自己]
- テストの実行結果（「全テスト通過」「○件成功」など）や、ビルドのステータスは記載を
  禁止する。[自己]
- `Co-Authored-By:` トレーラーは記載を禁止する。一部のモデルがコミットメッセージ末尾に
  自動付与することがあるため、含まれていれば削除すること。[機械]

### サンプル

```
#### 良い例（技術的な理由が書かれている）
- #123 の対応
- イベントフォーム画面で発生したランタイムエラーの対応
  * 引数の型が想定と異なっていたため、正しい型定義に変更

#### 悪い例（プロセスの経緯が書かれている）
- #123 の対応
- レビュー指摘の対応
  * 指摘された箇所の修正
```

## コミット実行方法

- 関連ファイルを個別に `git add` する（`git add -A` や `git add .` は使わない）。
- コミットメッセージは必ず HEREDOC 形式で渡す。`git commit -m` の引数に直接文字列を
  渡すと、改行が `\n` リテラルとして埋め込まれる問題があるため禁止する。

```bash
git commit -m "$(cat <<'EOF'
<コミット種別>: <コミットタイトル>

<コミット内容>
EOF
)"
```
