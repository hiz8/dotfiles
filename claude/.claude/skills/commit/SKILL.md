# commit: 規約に沿ったコミット

ローカルの変更を、表示幅ルールを含むコミットメッセージ規約に従ってコミットする。

## 手順

1. 以下のコマンドを並列実行して現在の状態を把握する:
   - `git status` で未追跡・変更ファイルを確認（`-uall` フラグは使わない）
   - `git diff` と `git diff --staged` でステージ済み・未ステージの変更を確認
   - `git log --oneline -10` で直近のコミットメッセージのスタイルを確認

2. すべての変更を分析して適切なコミット粒度を検討する:
   - ステージ済みのファイルが存在する場合は、それのみをコミット対象とする
   - 関連しない変更が混在していれば、コミットを分割する
   - シークレットを含む可能性のあるファイル（`.env`、認証情報など）はコミットしない

3. `git add` で関連ファイルを個別にステージする（`git add -A` や `git add .` は使わない）

4. コミットメッセージ規約 `references/commit-rules.md` を読み、テンプレートに沿って
   メッセージを作成する。特に次の点に注意する。
   - 1行目（subject）は `<種別>: <タイトル> <Issue番号>` 形式。種別は
     feat/fix/docs/refactor/test/chore。タイトルは50文字以内。
   - subject の後に空行を1行入れ、その後にコミット内容をリスト形式で書く。
   - 表示幅は全角2桁/半角1桁で80桁以内、かつ70〜80桁を目安に詰める
     （不必要に早い改行をしない）。

5. 作成したメッセージを一時ファイルに書き出し、機械チェックを実行する。

   ```bash
   node .claude/skills/commit/check-commit-message.mjs <message-file>
   ```

   エラーが出たら文面を再調整して再実行し、エラーが無くなる（exit 0）まで繰り返す。
   `--verbose` を付けると全行の表示幅一覧が出るので、桁数調整の参考にする。

   表示幅 70〜80 桁の目安と「早すぎる改行」の例外（文末で終わる行・次語を繰り上げると
   80 桁を超える行・URL/識別子を含む行）は、このチェックが判定する。70 桁未満の行が
   あっても、チェックがエラーにしない限り規約違反ではない。初回に「早すぎる改行」が
   出るのは想定内で、指摘された行に次行の語を繰り上げて詰め直せば解消する。逆に、
   チェックを通った行を手計算で「短い」と感じて再分割しないこと（過剰修正の防止）。

6. 検証を通過したら、HEREDOC 形式でコミットする。
   `git commit -m` の引数に直接文字列を渡すと改行が `\n` リテラルになるため禁止。
   メッセージ末尾に `Co-Authored-By:` トレーラーを付けないこと（機械チェックで弾かれる）。

   ```bash
   git commit -m "$(cat <<'EOF'
   <コミット種別>: <コミットタイトル> <Issue番号>

   <コミット内容>
   EOF
   )"
   ```

## 機械チェックで検証できないルール

`check-commit-message.mjs` は表示幅・タイトル長・種別プレフィックス・装飾記号のみを
検証する。次のルールは自動検証できないため、`references/commit-rules.md` を必ず読み、
コミット前に自己チェックすること。

- Why（技術的な原因・背景）を先に述べ、What を続ける。プロセス（「レビュー指摘の対応」
  など）を理由に書かない。
- 文末は体言止めまたは現在形。過去形・「です・ます」調は禁止。Why で既存の不備を
  述べる場合も過去形にせず、現在も続く状態として現在形・体言止めで書く
  （「〜していた」→「〜している／〜が生じる」）。
- テストの実行結果やビルドのステータスは記載しない。
- グルーピングは見出し記号でなくリストのネスト（インデント）で表現する。
