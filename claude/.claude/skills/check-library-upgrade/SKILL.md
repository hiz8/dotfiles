# ライブラリ更新内容の調査

指定されたライブラリについて、**現在プロジェクトで使用中のバージョン**と**指定バージョン**の間の更新内容を調査し、決められた4観点で日本語のレポートを作成する。

## 引数

`$ARGUMENTS` に `<ライブラリ名> <ターゲットバージョン>` の順で渡される。

- 第1引数 (`$1`): ライブラリ名 (npm パッケージ名)。例: `react`, `@tanstack/react-router`
- 第2引数 (`$2`): ターゲットバージョン。例: `19.2.0`, `1.180.0`。省略された場合は最新安定版を対象とする。最新安定版は `npm view <library> dist-tags` の `latest` を採用し、`beta` / `rc` / `next` / `canary` / `experimental` などのプレリリースタグは選ばない (`versions` 配列の末尾はプレリリースを含みうるため使わない)。

ライブラリ名が渡されていない場合は、何を調査すればよいかユーザーに確認してから進める。

## 手順

### 1. 現在使用中のバージョンを特定する

1. リポジトリ内の `package.json` を探す (ルートおよびワークスペース)。`Glob` で `**/package.json` を検索し、`node_modules` 配下は除外する。
2. 対象ライブラリが `dependencies` / `devDependencies` / `peerDependencies` のどれに、どの範囲 (`^4.1.0` など) で記載されているか確認する。
3. 実際にインストールされている確定バージョンを `npm ls <library> --depth=0` (取得できなければ lockfile) で確認する。範囲表記と確定バージョンが異なる場合は、確定バージョンを「現在のバージョン」として扱う。
4. 対象ライブラリが依存に存在しない場合はその旨を報告し、ユーザーに意図を確認する (新規導入の検討か、パッケージ名の誤りか)。このケースでは手順2以降に進まず、確認を提示した時点で停止する。いずれかの意図 (新規導入 / 名前誤り) を勝手に仮定して調査を続行してはならない。

### 2. パッケージとリリースノートの所在を特定する

1. `npm view <library> repository.url` と `npm view <library> homepage` でソースリポジトリ (多くは GitHub) を特定する。
2. `npm view <library> repository.directory` を確認し、モノレポ内のパッケージかどうか判断する (タグ名が `pkg@1.2.3` 形式になることがある)。
3. `npm view <library> versions --json` で、現在バージョンとターゲットバージョンの間に存在する全リリースを把握する。semver でメジャー跨ぎか (破壊的変更が入りやすい) を判断する。

### 3. リリースノートを収集する

現在バージョン (排他) からターゲットバージョン (包含) までの全リリースを対象に、変更内容を集める。以下を優先順に試す。

1. **GitHub Releases**: `WebFetch` で `https://github.com/<owner>/<repo>/releases` を取得。タグ命名の揺れ (`v1.2.3` / `1.2.3` / `pkg@1.2.3`) に注意する。
2. **CHANGELOG**: `WebFetch` で `https://raw.githubusercontent.com/<owner>/<repo>/<default-branch>/CHANGELOG.md` などを取得。
3. **移行ガイド**: メジャー跨ぎの場合、公式の migration / upgrade guide を `WebSearch` / `WebFetch` で探す。
4. 上記で不足する場合は `WebSearch` で補う。`mcp__context7__*` や GitHub の MCP ツールが利用可能な環境なら、補助的に活用してよい (必須ではない)。

各変更には、それが導入されたバージョンと出典 (リリースページ等) を必ず紐付ける。あわせて、収集した変更を「新機能の追加」と「それ以外 (バグ修正・パフォーマンス改善・内部リファクタ・ドキュメント変更・依存更新など)」に分類しておく。

**出典の優先順位 (一次情報の裁定)**: GitHub Release タグ本文・CHANGELOG を一次情報とする。`WebSearch` や要約を介した `WebFetch` の結果は二次情報として扱い、一次情報と一致する項目のみ採用する。一次情報に記載が無い変更は、検索で出てきてもレポートに載せない。一次情報と二次情報が食い違う場合は一次情報を優先する。一次情報自体が取得できない場合は、その旨を明記したうえで二次情報を「未確認」と注記して扱う。

### 4. プロジェクトへの影響を分析する (コード使用箇所まで)

リリースノートから抽出した「新機能」「破壊的変更」のそれぞれについて、プロジェクトの実コードを調べて影響を判定する。

- 破壊的変更が触れている API・関数・設定キー・import パス・型などを `Grep` でプロジェクト全体 (`node_modules` を除く) から検索する。
- 該当する使用箇所が見つかった破壊的変更を「影響あり」と判定し、`ファイルパス:行番号` を添えて具体的に示す。
- 新機能のうち、現在の手書き実装・回避策・非推奨 API の利用を置き換えられそうなものを「有用」と判定し、根拠となるコード箇所を示す。
- 判定は確証のあるものを優先する。推測が混じる場合はその旨を明示する。

### 5. レポートを作成する

以下の4セクション構成で、日本語のレポートを簡潔に出力する。各項目は要点を1〜2行にまとめ、導入バージョンと出典を添える。冗長な前置きや説明は省く。

**「なし」最小記法ルール (全セクション共通)**: 4つのセクションいずれも、該当が無ければ本文を `なし` の一語だけにする。理由・根拠・出典・「破壊的変更が無いことの説明」などの補足を**一切添えない** (セクション3・4のような重要セクションであっても同じ)。補足を書いてよいのは、レポート末尾の総括の一文のみ。

```
## <library> の更新調査: <現在バージョン> → <ターゲットバージョン>

対象リリース: <跨ぐバージョンの一覧 / メジャー跨ぎの有無>

### 1. 主な追加機能
- (バージョン間で**新しく追加された機能のみ**を、導入バージョンと出典付きで簡潔に列挙。バグ修正・パフォーマンス改善・内部リファクタ・ドキュメント変更・依存更新などは含めない)

### 2. 現在のプロジェクトに有用な追加機能
- (1 のうち、このプロジェクトのコードに照らして役立つもの。理由と該当コード箇所を添える。該当が無ければ「なし」とだけ書き、理由や補足説明は一切添えない)
- ここに載せてよいのは **1 に列挙した「追加機能」だけ**。バグ修正・小改善はプロジェクトのコードに接点があっても載せない (それらは自動適用され対応不要なため)。接点を見つけても「追加機能」でなければ「なし」と書く。無理に該当を作らない

### 3. 破壊的変更
- (バージョン間の破壊的変更を、導入バージョンと出典付きで列挙。該当が無ければ「なし」とだけ書き、理由や補足説明は一切添えない)
- ここでの「破壊的変更」は、既存のコードが**そのままでは動かなくなる**変更 (API の削除・改名・シグネチャ変更・デフォルト挙動の非互換変更など) を指す。**deprecation (非推奨化) は、旧 API が動作し続ける限りこのセクションに含めない**。後継 API があればセクション1で新機能として扱う

### 4. 現在のプロジェクトに影響がある破壊的変更
- (3 のうち、このプロジェクトのコードに実際の影響があるもの。該当する `ファイル:行` と必要な対応を添える。該当が無ければ「なし」とだけ書き、理由や補足説明は一切添えない)
```

最後に、3〜4 で重大な影響が見つかった場合は、対応の要否・規模を一文で総括する。

## 注意

- 調査の起点はあくまで「現在プロジェクトで使用中のバージョン」。最新版同士の差分ではない点に注意する。
- バージョン区間に複数のメジャー/マイナーが含まれる場合は、その全区間の変更を累積して扱う。
- リリースノートが見つからない・特定できない場合は、推測で埋めず「確認できなかった」と明記する。
- ネットワーク取得が制限されている環境では、取得できた範囲を明示したうえで報告する。
- レポートは簡潔さを優先する。各項目は要点を短くまとめ、列挙が長くなる場合は影響の大きいものに絞る。
